## 代码库模式
- vphoned 模块命名约定：文件 `vphoned_模块名.{h,m}`，函数 `vp_模块名_动词()`，命令处理 `vp_handle_模块_command()`
- Mach-O 检测使用 magic number（4 bytes header），支持 64-bit/32-bit/FAT 格式
- .app bundle 结构：主二进制通过 `Info.plist` 的 `CFBundleExecutable` 定位，框架二进制在 `Frameworks/*.framework/` 下
- iOS 私有 API 调用模式：`dlopen` 加载框架 → `NSClassFromString` 获取类 → `NSSelectorFromString` + `NSInvocation` 调用方法（避免编译期依赖）
- 私有框架加载双路径回退：先 `/System/Library/PrivateFrameworks/` 再 `/System/Library/Frameworks/`
- 头文件模板：`#pragma once` + `#import <Foundation/Foundation.h>` + 函数声明
- 响应构建：统一使用 `vp_make_response(type, reqId)` 返回 `NSMutableDictionary *`，错误时 type 传 `@"err"` 并追加 `r[@"msg"]`
- caps 注册模式：始终可用的写入初始数组，运行时探测的用 `if (vp_xxx_available())` 条件添加
- 命令分派：简单命令在 `handle_command()` 中用 `isEqualToString:` 匹配；需要 socket fd 的命令（如 file_*）在 `handle_client()` 消息循环中用 `hasPrefix:` 拦截
- vphoned 是 guest ObjC 代码，无法在 macOS 上编译验证（需 iOS SDK），语法检查依赖代码审查
- 外部命令执行使用 `posix_spawn` + `waitpid` 模式
- 所有 .m 文件只 import 自己的 .h 和 vphoned_protocol.h，不交叉 import 其他模块
- 宿主端高层 API 模式：`VPhoneControl` 中封装 struct（如 `InstallResult`）+ async throws 方法，内部组合 `uploadFile` + `sendRequest` 等底层调用
- `ControlError` 扩展模式：需要保留额外字段时新增 case（如 `guestErrorWithStage`），保持 `description` 返回用户可读字符串
- GUI 菜单扩展模式：`VPhoneMenuConnect` 中新增菜单项用 `makeItem` + `@objc` 方法；异步操作放在 `Task {}` 中，结果用 `showAlert` 展示
- 菜单项运行时禁用模式：通过 `NSApp.mainMenu?.item(withTitle:)?.submenu?.item(withTitle:)` 查找菜单项，设置 `isEnabled` 控制
- `onConnect` 回调有 GUI 和 headless 两处，修改连接后行为时两处都要同步更新
- CLI 参数添加模式：`VPhoneCLI.swift` 中用 `@Option`/`@Flag` 声明，`VPhoneAppDelegate` 通过 `cli.xxx` 访问

## 2026-03-03 - US-002
- 实现了 Guest 端可选 ldid 重签模块
- 修改文件：`scripts/vphoned/vphoned_app.h`、`scripts/vphoned/vphoned_app.m`
- 功能：`vp_app_resign()` 检测 ldid 可用性，收集 .app 内所有 Mach-O 二进制并调用 `ldid -S` 重签
- 辅助：`is_macho()` 通过 magic number 检测 Mach-O 文件，`collect_macho_binaries()` 遍历主二进制 + Frameworks
- **未来迭代的经验总结：**
  - `vp_app_resign()` 返回值语义：`YES` = ldid 可用且重签成功，`NO` = ldid 不可用（非错误）或重签失败（有 outError）
  - US-004 调用时需区分：返回 NO 且 outError == nil → ldid 不可用（resigned=false），返回 NO 且 outError != nil → 重签失败（stage=resign）
  - Mach-O magic 检测覆盖：MH_MAGIC_64/32 + big/little endian + FAT binary
  - framework 二进制查找：先读 Info.plist 的 CFBundleExecutable，回退到去掉 .framework 后缀的名称

## 2026-03-03 - US-003
- 实现了 Guest 端 LSApplicationWorkspace 安装调用
- 修改文件：`scripts/vphoned/vphoned_app.h`、`scripts/vphoned/vphoned_app.m`
- 功能：`vp_app_install()` 通过 dlopen + NSInvocation 调用 LSApplicationWorkspace 私有 API 安装 .app 包
- 辅助：`load_ls_workspace()` 双路径加载 CoreServices 框架，`rebuild_icon_cache()` 刷新 SpringBoard 图标缓存，`vp_app_cleanup_ipa()` 清理临时目录和 IPA 文件
- **未来迭代的经验总结：**
  - `vp_app_install()` 返回 bundleId（成功）或 nil（失败），US-004 用此构建响应中的 `bundle_id` 字段
  - 使用 `NSInvocation` 而非 `performSelector:` 调用多参数方法（含 NSError** 参数），这是 vphoned 中调用私有 API 的标准模式
  - `rebuild_icon_cache()` 有双重回退：`_LSPrivateRebuildApplicationDatabasesForSystemApps:internal:user:` → `/usr/bin/uicache -a`
  - `vp_app_cleanup_ipa()` 同时删除解压目录和源 IPA 文件，US-004 在安装完成后（无论成功失败）都应调用
  - extractDir 从 appPath 推导：`[[appPath stringByDeletingLastPathComponent] stringByDeletingLastPathComponent]`（即 `/tmp/<UUID>/Payload/X.app` → `/tmp/<UUID>/`）

## 2026-03-03 - US-004
- 实现了 Guest 端 app_install 命令分发与协议集成
- 修改文件：`scripts/vphoned/vphoned.m`、`scripts/vphoned/vphoned_app.h`、`scripts/vphoned/vphoned_app.m`
- 功能：`vp_handle_app_install()` 串联解压 → 可选重签 → 安装三个阶段，统一响应格式
- 集成：在 `handle_command()` 中添加 `app_install` 分支，在 caps 数组中注册 `@"app_install"` 能力
- **未来迭代的经验总结：**
  - `app_install` 是简单命令（不需要 socket fd），直接在 `handle_command()` 中处理（不像 `file_*` 需要在 `handle_client()` 中拦截）
  - caps 中 `app_install` 始终注册（不像 location 需运行时探测），因为底层 API 在安装时才 dlopen 加载，失败时会返回明确错误
  - US-005 宿主端需要：先 `uploadFile` 上传 IPA 到 `/tmp/install_<UUID>.ipa`，再 `sendRequest` 发送 `{"t": "app_install", "path": "/tmp/install_<UUID>.ipa"}`
  - 成功响应含 `bundle_id` + `resigned`，失败响应含 `msg` + `stage`（unzip/resign/install）
  - 每个阶段失败都会调用 `vp_app_cleanup_ipa()` 清理临时文件，确保无残留
